# IliaGPT

## Overview
IliaGPT is an AI-powered chat application designed as an intelligent assistant for autonomous web browsing and document creation. Its core purpose is to offer a versatile platform for AI-driven tasks, including economic data analysis, multi-intent prompt processing, and professional document generation. The ambition is for IliaGPT to become a leading AI assistant for productivity.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture
IliaGPT is built with a monorepo structure, separating client, server, and shared components, ensuring type safety with Zod schemas. The frontend uses React, TypeScript, Vite, shadcn/ui, and Tailwind CSS for a modern, themable interface with light/dark mode support. It features chat management, prompt templates, PWA, keyboard shortcuts, and renders Markdown, code, and mathematical expressions. Data visualization is handled by Recharts, ECharts, and TanStack Table, supporting SVG, Canvas 2D, and 3D. Performance is optimized with virtualization, memoization, lazy loading, and streaming UX. Security includes DOMPurify, frontend rate limiting, and MIME type validation. A file preview system supports various document types.

The backend uses Node.js and Express.js, featuring an LLM Gateway for AI model interactions with multi-provider fallback, caching, token tracking, and circuit breakers. Agentic tool calling is supported, including OpenAI-compatible function calling and an ETL Agent for economic data. A Multi-Intent Pipeline handles complex prompts, and a PARE System processes documents with per-document citations. A Document Generation System uses LLM orchestration for Excel and Word files. The Spreadsheet Analyzer Module provides AI-powered analysis and a secure Python sandbox.

Integrated agentic tools from OpenClaw v2026.3.1 include `bash` (hardened), `web_fetch`, `web_search` (DuckDuckGo), file operations, `run_code` (isolated Python/Node), process management, and RAG indexing. The agent executor uses OpenAI-compatible API calls via OpenRouter.

**Auth Storage**: `email_canonical` column added to `users` table schema for email dedup/canonicalization. Auth identity resolution uses a 4-step upsert: (1) provider identity lookup, (2) user ID match, (3) canonical email match, (4) new user creation. Google OAuth Passport callback uses defensive null-safe extraction of profile fields before calling `upsertUser`. Auto-migration (`ensureAuthColumns`) adds missing columns (`email_canonical`, `org_id`, `login_count`, `full_name`, `profile_image_url`) at startup. **Google OAuth session fix**: `state: true` maintained (CSRF protection). Session pre-dirty (`__oauth_init` marker) on `/api/auth/google` route ensures `saveUninitialized: false` doesn't skip state persistence. Cookie `sameSite: "lax"` (not `"none"` â€” lax works for OAuth top-level navigations and avoids Safari ITP issues). Diagnostic endpoint: `GET /api/auth/google/check` (admin-only, shows callbackURL, baseUrl, detected origin). Detailed error logging in callback with `google_state_mismatch` error differentiation. Callback logs redact OAuth codes (only log `hasCode`/`hasState` booleans).

**Agentic-First Architecture**: `isAgenticMode` is always `true` â€” every message goes through `executeAgentLoop` with full tool access; the non-agentic `llmGateway.streamChat` direct path is eliminated. Tool forcing (`tool_choice: "required"`) on iterations 1-2 is now intent-aware: only forces when `intent !== "chat"` or message length > 30 chars, preventing forced tool calls on simple greetings. Agentic nudge similarly gated by `isActionIntent`. Default model: `google/gemini-2.5-flash`. Execution plan SSE events: `plan` (emitted before loop with intent-specific steps) and `exec_plan_update` (emitted per iteration and on completion) provide structured progress tracking. Inference progress interval emits `thinking` events every 2.5s during LLM inference wait ("Analizando tu solicitud..." â†’ "Evaluando herramientas..." â†’ "Decidiendo plan..." â†’ "Preparando ejecuciĂłn...") to prevent UI freezing. Frontend handles `plan`, `exec_plan_update`, and inference `thinking` events with proper state management â€” inference steps replace each other, plan steps show pending/active/done status.
